The Sorting Hat: How Trades Get Grouped

The hardest problem in the app: taking hundreds of individual fills and figuring out which ones belong together as a single round-trip trade. Here's the exact code that does it — from trading_journal/trade_completion.py:

CODE 
trade_groups: Dict[Any, List[Trade]] = {}
for trade in unlinked_trades:
    key = (trade.symbol, trade.instrument_type, trade.account_id)
    if trade.instrument_type == 'OPTION' and trade.option_data:
        key = key + (trade.exp_date, trade.strike_price, trade.option_type)
    elif trade.instrument_type == 'FUTURES':
        key = key + (trade.exp_date,)
    trade_groups.setdefault(key, []).append(trade)

completed_count = 0
for trades in trade_groups.values():
    completed_count += self._process_trade_group(session, trades)
PLAIN ENGLISH

trade_groups = {} — Create an empty set of buckets. Each bucket will hold fills that belong together. The bucket's label is a combination of identifying details.

for trade in unlinked_trades: — Loop through every fill that hasn't been matched to a completed trade yet.

key = (symbol, instrument_type, account_id) — Every fill gets sorted by: what ticker it is, what kind of asset (stock vs instrument type), and which brokerage account it came from. AAPL stock in Account A never mixes with AAPL stock in Account B.

if OPTION: extend key with exp_date + strike_price + option_type — Options need more precision. A $200 call expiring in January is a completely different contract from a $200 call expiring in March. The expiration date, strike price, and call-vs-put type are all added to the bucket label.

elif FUTURES: extend with exp_date — Futures contracts also have expiration dates that make them distinct.

trade_groups.setdefault(key, []).append(trade) — Drop this fill into the matching bucket. If no bucket exists for this key yet, create one first.

for trades in trade_groups.values(): — Now process each bucket. Every bucket is a set of fills that all belong to the same trade.

_process_trade_group(session, trades) — Hand the bucket to the method that scans through the fills chronologically and seals completed round-trips when the position returns to zero shares.

First Day on the Job

The six classes introduce themselves in the team workspace. Click through to meet everyone.

📋
CsvParser

Hey team! I'm CsvParser. I handle everything about reading your broker's file format — column name quirks, the account number buried on line 1, options section headers. Format weirdness? My problem, not yours. You give me a CSV; I give you clean records.

🔒
NdjsonIngester

NdjsonIngester here. I'm the gatekeeper. Every record gets validated by Pydantic before it touches the database — wrong data type, missing field, invalid symbol? I reject it cleanly. I also handle the unique_key deduplication so re-uploads are always safe.

🔗
TradeCompletionEngine

TradeCompletionEngine. I find the buy-sell pairs. I sort every fill into buckets — by symbol, account, and (for options) the specific contract. When a bucket's position returns to zero, I seal it as a completed trade. That's the core problem I exist to solve.

📐
PositionTracker

I'm PositionTracker. I do the math — average cost basis, realized P&L, unrealized P&L. I also apply contract multipliers — one options contract is worth 100 shares, not 1. Get that wrong and every P&L number is garbage.

📊
DashboardEngine

DashboardEngine! I crunch your analytics — win rate, profit factor, average winner, average loser, longest win streak. I'm strictly read-only though. I analyze data, never write it. Think of me as the analyst who reads everyone else's reports.

📡
MassiveClient

MassiveClient here. I'm the outside contractor. When you need historical price bars from Polygon.io — for enriching trades with market context or running the HG plan analysis — you call me. I'm not in the main upload pipeline; I get called on-demand.

📋
CsvParser

Great team! One thing worth noting: we each own exactly one concern. If you ever find yourself thinking "this class does too many things" — that's the signal to split it. That's what engineers call "separation of concerns."

Separation of Concerns

Each class knows how to do one thing and trusts the others. This is the most important pattern in software engineering — and the reason you can tell AI "modify PositionTracker" instead of "change the math stuff."
