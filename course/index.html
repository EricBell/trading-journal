The Fingerprint That Prevents Duplicates

Upload the same file twice? Nothing bad happens. The app uses a unique_key as a fingerprint for each trade. Here's exactly how it's built — straight from trading_journal/schemas.py:

CODE 
@property
def unique_key(self) -> str:
    """Generate unique key for upsert logic."""
    # Use source file, row index, and key trade details
    base_key = f"{self.source_file}:{self.row_index}"

    if self.exec_time:
        base_key += f":{self.exec_time.isoformat()}"
    elif self.time_canceled:
        base_key += f":{self.time_canceled.isoformat()}"

    if self.symbol:
        base_key += f":{self.symbol}"

    if self.side and self.qty:
        base_key += f":{self.side}:{self.qty}"

    return base_key
PLAIN ENGLISH

@property — This is a computed attribute. You read it like record.unique_key and Python runs this code to generate the value on the spot.

base_key = filename:row_number — Start the fingerprint with two things we already know: which file this came from, and which row in that file. This alone is often enough to identify a record.

if exec_time: append timestamp — If we have the exact time the trade was executed, glue it on. A timestamp makes the fingerprint even harder to fake-duplicate.

elif time_canceled: … — Some records are canceled orders, not fills. Use their cancellation time as the timestamp instead.

if symbol: append it — Add the ticker symbol (AAPL, TSLA, etc.) to the fingerprint.

if side and qty: append them — Finish with BUY or SELL and the number of shares. Now the fingerprint encodes: file + row + time + symbol + direction + size. Two rows with all the same values are the same trade. One difference anywhere = a different fingerprint.

return base_key — Hand back the finished fingerprint string. The UPSERT logic uses this to decide: insert new row or update existing one?

💡
This Makes the App Idempotent

Re-upload any file as many times as you want. The unique_key system guarantees you never get duplicate trades.

The Sorting Hat: How Trades Get Grouped

The hardest problem in the app: taking hundreds of individual fills and figuring out which ones belong together as a single round-trip trade. Here's the exact code that does it — from trading_journal/trade_completion.py:

CODE 
trade_groups: Dict[Any, List[Trade]] = {}
for trade in unlinked_trades:
    key = (trade.symbol, trade.instrument_type, trade.account_id)
    if trade.instrument_type == 'OPTION' and trade.option_data:
        key = key + (trade.exp_date, trade.strike_price, trade.option_type)
    elif trade.instrument_type == 'FUTURES':
        key = key + (trade.exp_date,)
    trade_groups.setdefault(key, []).append(trade)

completed_count = 0
for trades in trade_groups.values():
    completed_count += self._process_trade_group(session, trades)
PLAIN ENGLISH

trade_groups = {} — Create an empty set of buckets. Each bucket will hold fills that belong together. The bucket's label is a combination of identifying details.

for trade in unlinked_trades: — Loop through every fill that hasn't been matched to a completed trade yet.

key = (symbol, instrument_type, account_id) — Every fill gets sorted by: what ticker it is, what kind of asset (stock vs instrument type), and which brokerage account it came from. AAPL stock in Account A never mixes with AAPL stock in Account B.

if OPTION: extend key with exp_date + strike_price + option_type — Options need more precision. A $200 call expiring in January is a completely different contract from a $200 call expiring in March. The expiration date, strike price, and call-vs-put type are all added to the bucket label.

elif FUTURES: extend with exp_date — Futures contracts also have expiration dates that make them distinct.

trade_groups.setdefault(key, []).append(trade) — Drop this fill into the matching bucket. If no bucket exists for this key yet, create one first.

for trades in trade_groups.values(): — Now process each bucket. Every bucket is a set of fills that all belong to the same trade.

_process_trade_group(session, trades) — Hand the bucket to the method that scans through the fills chronologically and seals completed round-trips when the position returns to zero shares.

The Three-Tier Design

How raw fills become P&L — and why everything except the raw data can be destroyed and rebuilt.

The Ship Navigation Metaphor

The database has 13 tables, but only three of them form the spine of the whole system. Everything else supports this central architecture.

📡

Raw GPS Log

Every position fix, every second. Never deleted. Never modified. The only record that matters when everything else fails.

🗺️

Plotted Route Summary

Departure → arrival, distance, speed. Derived from the raw log. Can be regenerated at any time. Deleted and rebuilt when the route changes.

🧭

Current Position Indicator

Where you are right now. Also derived. Updated continuously from the raw log. Always current, never the original source.

💡
The Core Principle

The raw log is the only thing you must never lose. The summaries and current position can always be regenerated from it. This is exactly how the Trading Journal database works — and it's one of the most important patterns in data engineering.

The Three Tiers

Each tier is built from the one below it. Tiers 2 and 3 are fully derived. The only irreplaceable data is Tier 1.

Tier 3 · Positions Fully derived · 1 row per symbol

The positions table. Running totals — how many shares you own right now and what you paid on average. Rebuilt from scratch whenever fills change.

current_qty avg_cost_basis realized_pnl
↑ derived from ↓
Tier 2 · Completed Trades Fully derived · 1 row per round-trip

The completed_trades table. Each buy→sell round trip. Entry price, exit price, net P&L, timestamps. Destroyed and rebuilt on every file upload.

entry_avg_price exit_avg_price net_pnl opened_at closed_at
↑ derived from ↓
Tier 1 · Trades (Executions) 🔒 Source of truth · Never deleted

The trades table. One row per broker fill. Raw, immutable, the GPS log. Every P&L number in the app ultimately comes from this table.

unique_key exec_timestamp symbol side qty net_price pos_effect

The Survivor: trade_annotations

There is a fourth table that sits beside the three tiers — not derived from Tier 1. This is trade_annotations: the setup patterns, notes, and stop prices you manually typed in. It lives in its own table with its own survival strategy. More on this in a moment.

CODE 
def reprocess_all_completed_trades(self, user_id):
  with self.db_manager.get_session() as session:
    # Unlink all executions from their completed trades
    session.query(Trade).filter(
      Trade.user_id == user_id
    ).update(
      {Trade.completed_trade_id: None},
      synchronize_session=False
    )
    # Delete all completed trades for this user
    session.query(CompletedTrade).filter(
      CompletedTrade.user_id == user_id
    ).delete(synchronize_session=False)
    session.commit()
PLAIN ENGLISH

Open a database session (a transaction that either fully completes or fully rolls back).

First, sever all the links from individual fills back to their round-trip trade rows — set those foreign key pointers to NULL.

Then delete every completed_trade row for this user. Tier 2 is now empty.

Commit — make it permanent. Tier 1 (the fills) is untouched throughout. Only the derived data is gone.

How Your Notes Survive the Rebuild

When Tier 2 is deleted, the foreign key from trade_annotations to completed_trades is set to NULL (the pointer goes dead). After the rebuild, this SQL reconnects everything.

CODE 
# Re-link orphaned annotations after the rebuild.
# Natural key: (user_id, symbol, opened_at)
session.execute(
  text("""
    UPDATE trade_annotations ta
    SET completed_trade_id = ct.completed_trade_id
    FROM completed_trades ct
    WHERE ta.user_id   = :user_id
      AND ta.completed_trade_id IS NULL
      AND ct.user_id   = :user_id
      AND ta.symbol    = ct.symbol
      AND ta.opened_at = ct.opened_at
  """),
  {"user_id": user_id},
)
PLAIN ENGLISH

The rebuild is done — Tier 2 now has fresh completed_trade_id numbers. Time to reconnect the annotations.

UPDATE trade_annotations: For every annotation row that currently has a dead/NULL link to a completed trade...

SET completed_trade_id: ...point it to the newly-created row in the fresh Tier 2...

WHERE ... IS NULL: Only fix annotations that lost their link (the ones with NULL). Leave already-connected ones alone.

AND ta.symbol = ct.symbol AND ta.opened_at = ct.opened_at: Match by symbol + timestamp. Like reuniting luggage with its owner by matching name, destination, and flight number — three facts that never change.

This is the natural key pattern: even when IDs are recycled, (user, symbol, opened_at) uniquely identifies the trade forever.

🔗
ON DELETE SET NULL

The ON DELETE SET NULL constraint is what makes this safe. When completed_trades rows are deleted during the rebuild, the database automatically sets all referencing completed_trade_id fields to NULL rather than deleting the annotation along with them.

Vocabulary Is Power

If you can say "use an idempotent UPSERT keyed on (user_id, unique_key)" instead of "make it not create duplicates," AI will write better code, faster.

Pattern 1: Idempotent UPSERT

Every time a fill is ingested, the app uses PostgreSQL's ON CONFLICT to turn a potential duplicate into an update. This is the full pattern from the real file.

CODE 
# Use PostgreSQL UPSERT
stmt = insert(Trade).values(**trade_data)
stmt = stmt.on_conflict_do_update(
    index_elements=['user_id', 'unique_key'],
    set_=dict(
        exec_timestamp=stmt.excluded.exec_timestamp,
        net_price=stmt.excluded.net_price,
        realized_pnl=stmt.excluded.realized_pnl,
        account_id=stmt.excluded.account_id,
        processing_timestamp=stmt.excluded.processing_timestamp
    )
).returning(Trade.trade_id)
PLAIN ENGLISH

insert(Trade).values(...): Try to INSERT a new row into the trades table with all the fill data.

on_conflict_do_update(...): But if this exact (user_id + unique_key) combination already exists — don't create a duplicate. Instead, do an UPDATE.

index_elements=...: The fingerprint for "does this already exist?" is the combination of whose account it is AND the unique identifier of the fill.

set_=dict(...): When updating, refresh these specific fields with the new values from the file — in case prices or timestamps were corrected.

.returning(Trade.trade_id): Either way — new row or updated row — give back the trade ID so the rest of the code knows which row it is working with.

Without UPSERT

Upload same CSV twice → 2 copies of every fill → P&L is double-counted → account looks twice as profitable (or twice as bad).

With UPSERT

Upload same CSV 10 times → still exactly 1 copy of every fill → numbers are always accurate regardless of how many times a file is re-imported.

Pattern 2: Symbol-Scoped Reprocessing

This fix was born from a real production crisis. Before it existed, uploading a file caused the server to crash. Here is the code, then the story.

CODE 
# Reprocess positions only for symbols present in this batch.
# Scoping to affected symbols avoids rebuilding the entire
# position history on every upload (which caused worker
# timeouts with remote PostgreSQL).
affected_symbols = {r.symbol for r in records if r.symbol}
self.position_tracker.reprocess_positions_for_symbols(
    user_id, affected_symbols
)
PLAIN ENGLISH

affected_symbols = {r.symbol for ...}: Build a set of every distinct stock symbol in this batch of records — e.g., {"AAPL", "TSLA", "SPY"}. Duplicates are automatically removed because it is a set.

reprocess_positions_for_symbols(...): Rebuild positions ONLY for those 3 symbols — completely skip the other 200+ symbols already in your history.

This is called symbol-scoped reprocessing. The comment in the code says it all — without this, remote database calls caused a server timeout that killed user uploads.

Average Cost Basis

When you buy in multiple batches, no single fill price is "your price." The app blends them.

CODE 
trade_qty = trade.qty
multiplier = get_contract_multiplier(
    trade.instrument_type, trade.symbol or ''
)
trade_cost = (Decimal(str(trade.net_price))
              * abs(trade_qty) * multiplier)

if trade.side == "SELL":
    trade_qty = -trade_qty  # Short position
PLAIN ENGLISH

trade_qty: Start with the raw quantity from the broker fill.

get_contract_multiplier(...): Look up how many "real dollars" one unit represents. For stocks: 1. For options: 100. For futures: depends on the contract.

trade_cost = Decimal(...): Calculate the total dollar cost of this fill using precise Decimal math — never regular floats for money.

if SELL, trade_qty = -trade_qty: Short positions are represented as negative quantities. Selling 100 shares becomes -100 to reduce the position count.

Contract Multiplier Lookup

A fill price of "$3.50" means very different dollar amounts depending on what instrument you traded.

CODE 
def get_contract_multiplier(instrument_type, symbol=''):
    if instrument_type == 'OPTION':
        return Decimal('100')
    if instrument_type == 'FUTURES':
        root = symbol.split()[0].upper() if symbol else ''
        return FUTURES_POINT_VALUES.get(root, Decimal('1'))
    return Decimal('1')
PLAIN ENGLISH

if OPTION → return 100: Every option contract represents 100 shares of the underlying stock. A $3.50 premium × 100 = $350 actual exposure per contract.

if FUTURES → look up by root symbol: Extract the base contract name (e.g., "MES" from "MES MAR25") and look it up in a table. MES (Micro E-mini S&P) = $5 per point. ES (E-mini S&P) = $50 per point.

return Decimal('1'): Stocks and ETFs are 1:1. A fill price of $150 per share means $150 per share. No conversion needed.

EQUITY / ETF Multiplier: 1× — fill price is actual dollar value per share
OPTION Multiplier: 100× — one contract = 100 shares. $3.50 premium = $350 per contract
MES (Micro E-mini S&P) Multiplier: $5/point — a 10-point move = $50 per contract
ES (E-mini S&P) Multiplier: $50/point — a 10-point move = $500 per contract
MNQ (Micro Nasdaq) Multiplier: $2/point — scaled-down version of NQ ($20/point)

The Four Known Limitations

Every codebase has known rough edges. The best engineers document them honestly. Here are the four in this app — understanding them helps you ask AI to solve the right problem, not work around the wrong one.

⏱️

Full Rebuild Bottleneck

TradeCompletionEngine rebuilds all completed trades for a user on every CSV upload — not just the new ones. Fine now. Will slow down at scale.

Fix direction: scope to affected symbols, like PositionTracker already does.

🏦

Single Broker Format

CsvParser only understands Schwab CSV. Supporting a new broker (TD Ameritrade, IBKR) means writing a new parser class.

Fix direction: new parser outputs same NdjsonRecord schema — rest of pipeline is already format-agnostic.

📡

No Live Data

Everything is file-import only. There is no live broker connection. Historical OHLCV bars come from Polygon.io on demand — not in real time.

Fix direction: requires a broker API integration — a separate, substantial project.

🔗

Annotation Re-linking is Best-Effort

trade_annotations re-links to rebuilt trades using (user_id, symbol, opened_at). If two trades for the same symbol open at the exact same millisecond, one annotation might link to the wrong trade.

Extremely rare in practice. Not worth fixing until it causes a real problem.

🧭
Navigator tip

When AI suggests a fix for one of these limitations, ask it to explain how the fix handles the edge case. If it cannot, it may not understand the constraint fully.
